1.1 Three-Level Namespace Architecture[a]
Unity Catalog extends the traditional two-level namespace with a top-level catalog layer, creating a three-part naming convention for every asset:
catalog.schema.table_or_view
This hierarchy enables centralized governance across workspaces and environments while providing flexible isolation boundaries.

Catalog Naming Rules
* Use lowercase with underscores as separators
* Keep names under 30 characters
* Avoid special characters except underscores

Part 2: Lakebase Branching Design for Enterprise
2.1 Why Database Branching Matters
Database branching brings Git-like workflows to database management. Instead of maintaining separate database instances for each environment, teams create lightweight copy-on-write branches that share underlying storage and diverge only where changes occur.
The Lakebase Foundation, Inspired by Neon
Databricks acquired Neon in May 2025, bringing copy-on-write PostgreSQL branching to the Databricks platform as Lakebase Autoscaling. Key architectural principles inherited from Neon:


Principle
	Description
	Copy-on-Write (CoW)
	Branches share base storage; only modified pages are stored separately
	Instant creation
	Branch creation is O(1) metadata operation regardless of database size
	Zero impact on parent
	Creating a branch does not affect the parent branch performance
	Scale-to-zero
	Idle branch compute suspends automatically, eliminating waste
	Point-in-time branching
	Create branches from any historical timestamp within the restore window
	Industry Validation
Database branching is now a production practice across the ecosystem:


Platform
	Engine
	Branching Approach
	Neon
	PostgreSQL
	Page-level CoW, instant, schema+data
	Lakebase
	PostgreSQL (Neon)
	Page-level CoW with Lakehouse integration
	2.2 Lakebase Autoscale Branching Capabilities
Branch Hierarchy[c][d]
Every Lakebase project starts with a default production branch. The hierarchy supports flexible parent-child relationships:
production (default, protected)
  |-- staging (protected)
  |     |-- development
  |     |     |-- dev/alice (TTL: 7d)
  |     |     |-- dev/bob (TTL: 7d)
  |     |     |-- ci/pr-142 (TTL: 4h)
  |     |     |-- ci/pr-155 (TTL: 4h)
  |     |-- qa/release-3.2 (TTL: 14d)
  |-- hotfix/critical-fix (TTL: 24h)


Branch Types


Type
	Behavior
	Use Case
	Default Branch
	Always-on, cannot be deleted, exempt from compute limits
	Production workloads
	Protected Branch
	Cannot be deleted, reset, or auto-archived
	Staging, critical environments
	Ephemeral Branch
	Has TTL (max 30 days), auto-deletes on expiry
	CI/CD, feature development
	Point-in-Time Branch
	Created from historical timestamp within restore window (0-30 days)

	Branch Operations via CLI
# Create a branch
databricks postgres create-branch projects/my-project development \
  --json '{"spec": {"source_branch": "projects/my-project/branches/production", "ttl": "604800s"}}'


# List all branches
databricks postgres list-branches projects/my-project --output json


# Get branch details
databricks postgres get-branch projects/my-project/branches/development


# Reset branch from parent
databricks postgres reset-branch projects/my-project/branches/development


# Protect a branch
databricks postgres update-branch projects/my-project/branches/production spec.is_protected \
  --json '{"spec": {"is_protected": true}}'


# Delete a branch
databricks postgres delete-branch projects/my-project/branches/development


Branch Operations via Python SDK
from databricks.sdk import WorkspaceClient
from databricks.sdk.service.postgres import Branch, BranchSpec, Duration


w = WorkspaceClient()


# Create branch with TTL
branch = w.postgres.create_branch(
    parent=f"projects/{project_name}",
    branch=Branch(spec=BranchSpec(
        source_branch=prod_branch_name,
        ttl=Duration(seconds=86400)
    )),
    branch_id="feature-branch"
).wait()


# Delete branch
w.postgres.delete_branch(
    name="projects/my-project/branches/feature-branch"
).wait()

	2.5 Enterprise Branching Patterns
Pattern 1: Simple Dev/Prod (Small Teams)
Best for teams of 1-5 developers with a single application.
production (protected)
  |-- development


* Two branches: production and development
* Development resets from production weekly
* Migrations tested on development, replayed on production
Pattern 2: Multi-Environment Pipeline (Medium Teams)
Best for teams of 5-20 with formal release processes.
production (protected)
  |-- staging (protected)
  |     |-- development
  |-- hotfix-ticket (ephemeral, TTL: 24h)


* Three tiers: production, staging, development
* Staging is protected; development resets from staging
* Hotfix branches created directly from production for urgent fixes
Pattern 3: Per-Developer Isolation (Large Teams)
Best for teams of 20+ working on the same database schema.
production (protected)
  |-- staging (protected)
  |     |-- development
  |           |-- dev-alice (TTL: 7d)
  |           |-- dev-jordan (TTL: 7d)
  |           |-- dev-sam (TTL: 7d)


* Each developer gets an isolated branch
* Personal branches reset from development when starting new work
* Schema Diff validates changes before promotion
Pattern 4: CI/CD Ephemeral Branches (DevOps Teams)
Best for teams with mature CI/CD pipelines.
production (protected)
  |-- staging (protected)
  |     |-- ci-pr-123 (TTL: 4h)
  |     |-- ci-pr-124 (TTL: 4h)
  |     |-- ci-pr-125 (TTL: 4h)


* Each pull request gets its own database clone
* TTL handles automatic cleanup
* Eliminates flaky shared test databases
Pattern 5: Multi-Tenant (Project-Level Isolation)
Best for SaaS platforms requiring strict tenant isolation.
Project: tenant-alpha
  |-- production (protected)
  |     |-- development


Project: tenant-beta
  |-- production (protected)
  |     |-- development


* Separate Lakebase projects per tenant
* Each project has its own branch hierarchy, compute, and storage
* Unity Catalog enforces cross-tenant access control
Pattern 6: Multi-Tenant (Schema-Level Isolation)
Best for cost-sensitive multi-tenant applications.
Project: multi-tenant-app
  |-- production (protected)
  |     |-- development


PostgreSQL Schemas:
  - tenant_alpha (with RLS policies)
  - tenant_beta (with RLS policies)
  - shared (common reference data)


* Single project, multiple PostgreSQL schemas
* Row-Level Security (RLS) enforces tenant isolation
* Data API enforces RLS automatically via Databricks identity
2.6 Schema Migration Workflow with Branching
The Safe Migration Testing Workflow
Step 1: Developer writes migration files locally (using Flyway, Liquibase, Prisma, or plain SQL)
Step 2: PR is opened, triggering CI/CD pipeline
Step 3: Pipeline creates a Lakebase branch from staging:
databricks postgres create-branch projects/my-project ci-pr-142 \
  --json '{"spec": {"source_branch": "projects/my-project/branches/staging", "ttl": "14400s"}}'


Step 4: Migrations are applied to the branch
Step 5: Schema Diff is captured (via Lakebase App UI or API)
Step 6: Integration tests run against the migrated branch
Step 7: Code review includes both code changes AND schema diff
Step 8: On PR merge, migrations are replayed on staging, then production
Step 9: Branch auto-deletes after TTL (or explicitly on PR close)
Idempotent DDL Best Practices
Since Lakebase does not support merge, all migrations must be idempotent (safe to re-run):
-- Good: Idempotent DDL
CREATE TABLE IF NOT EXISTS patients (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);


ALTER TABLE patients ADD COLUMN IF NOT EXISTS email VARCHAR(255);


CREATE INDEX IF NOT EXISTS idx_patients_email ON patients(email);


-- Bad: Non-idempotent DDL (will fail on re-run)
CREATE TABLE patients (...);
ALTER TABLE patients ADD COLUMN email VARCHAR(255);


2.7 GitHub Actions for Lakebase Branch Automation
Create Branch on PR Open
name: Create Lakebase Branch
on:
  pull_request:
    types: [opened, reopened]
jobs:
  create-branch:
    runs-on: ubuntu-latest
    steps:
      - name: Install Databricks CLI
        run: curl -fsSL https://raw.githubusercontent.com/databricks/setup-cli/main/install.sh | sh


      - name: Create Lakebase Branch
        env:
          DATABRICKS_HOST: ${{ secrets.DATABRICKS_HOST }}
          DATABRICKS_TOKEN: ${{ secrets.DATABRICKS_TOKEN }}
        run: |
          databricks postgres create-branch projects/${{ vars.LAKEBASE_PROJECT }} ci-pr-${{ github.event.pull_request.number }} \
            --json '{"spec": {"source_branch": "projects/${{ vars.LAKEBASE_PROJECT }}/branches/staging", "ttl": "14400s"}}'


      - name: Run Migrations
        run: |
          # Apply migrations against the branch endpoint
          flyway -url=jdbc:postgresql://$BRANCH_ENDPOINT:5432/mydb migrate


      - name: Run Tests
        run: pytest tests/integration/ --db-branch=ci-pr-${{ github.event.pull_request.number }}


Delete Branch on PR Close
name: Delete Lakebase Branch
on:
  pull_request:
    types: [closed]
jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - name: Delete Lakebase Branch
        env:
          DATABRICKS_HOST: ${{ secrets.DATABRICKS_HOST }}
          DATABRICKS_TOKEN: ${{ secrets.DATABRICKS_TOKEN }}
        run: |
          databricks postgres delete-branch \
            projects/${{ vars.LAKEBASE_PROJECT }}/branches/ci-pr-${{ github.event.pull_request.number }} || true

3.3 Enterprise Governance Checklist
Unity Catalog Governance
* Assign production catalog ownership to groups, not individuals
* Grant USE CATALOG and USE SCHEMA only to authorized users
* Implement row-level filtering and column masking for sensitive data
* Use deployment scripts (Terraform, DABs) for privilege assignments
* Audit permissions regularly using system.access.audit
* Bind catalogs to specific workspaces for isolation
* Document all catalogs and schemas with tags and comments
Lakebase Governance
* Mark production and staging branches as protected
* Enforce TTL policies on all non-protected branches
* Use CI/CD automation for branch creation and deletion
* Validate all schema changes via Schema Diff before promotion
* Write idempotent DDL for all migrations
* Monitor branch count (max 10 unarchived per project)
* Use Databricks identity (OAuth) instead of password-based access
* Enable Row-Level Security for multi-tenant applications
* Configure the Data API with appropriate exposed schemas and CORS origins
Combined Framework
* Align Lakebase project names with Unity Catalog domain names
* Use consistent environment prefixes across both systems (accounting for separator differences)
* Centralize access control through Unity Catalog groups
* Track data lineage from Lakebase OLTP to Delta Lake analytics
* Implement a single CI/CD pipeline that manages both Lakebase branches and Unity Catalog permissions

Part 4: Neon-Inspired Patterns for Lakebase 
4.1 Patterns Available Today
These patterns are directly implementable with current Lakebase Autoscaling capabilities.
Branch-per-Pull-Request
Inspired by Neon's most popular CI/CD pattern. Each PR gets an isolated database clone:
PR opened  ->  Create Lakebase branch (ci-pr-142, TTL: 4h)
               -> Run migrations on branch
               -> Run integration tests
PR merged  ->  Replay migrations on staging/production
               -> Branch auto-deletes after TTL
PR closed  ->  Branch auto-deletes after TTL


Branch-per-Developer
Each engineer gets a persistent isolated environment that resets periodically:
Developer onboard  ->  Create branch: dev-alice (TTL: 7d, auto-renew)
                       -> Alice develops against her own branch
                       -> Weekly reset from development branch
Developer offboard ->  Delete branch


Point-in-Time Audit Branches
Create read-only snapshots for compliance investigations:
Audit request  ->  Create branch from timestamp: audit-2026-02-21
                   -> Auditors query historical state
                   -> Branch auto-deletes after 30 days


4.2 Patterns Requiring Custom Development
These patterns require building automation on top of Lakebase primitives.
Schema Diff in PR Comments
Neon posts schema diffs directly to GitHub PR comments. For Lakebase, build a custom GitHub Action:
1. Create branch from staging
2. Apply migrations
3. Use Lakebase Schema Diff API (or pg_dump comparison) to generate diff
4. Post diff as a PR comment using GitHub API
5. Clean up branch
Automated Nightly Reset
Lakebase supports scheduled branch resets, automate with Databricks Jobs:
# Databricks Job: Nightly Staging Reset
# Schedule: 0 2 * * * (2 AM daily)
# Notebook content:
from databricks.sdk import WorkspaceClient
w = WorkspaceClient()
w.postgres.reset_branch(name="projects/my-project/branches/staging")


Branch Observability Dashboard
Build a monitoring dashboard tracking:
* Active branch count per project
* Branch age distribution
* Branch storage consumption (divergence from parent)
* Branch creation/deletion rate
* TTL compliance (branches approaching expiration)
4.3 AI Agent Database Branching
An emerging pattern from the industry: AI coding agents create database branches for safe testing.
Implementation for Lakebase
Store branching instructions in your project's CLAUDE.md or AGENTS.md:
## Database Testing Instructions


When making database schema changes:
1. Create a Lakebase branch: databricks postgres create-branch projects/my-project ai-agent-test --json '{"spec": {"source_branch": "projects/my-project/branches/development", "ttl": "3600s"}}'
2. Wait for branch to be ACTIVE
3. Apply and test migrations against the branch endpoint
4. Validate with Schema Diff
5. On success, provide the migration SQL for human review
6. Branch auto-deletes after 1 hour


This enables every AI-generated schema change to be tested against a production-shaped database without risk.
